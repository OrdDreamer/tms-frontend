# Готовність до React Compiler

Проєкт орієнтований на **React 19** і можливість увімкнення **React Compiler** без масового переписування. Ідея: **не покладатися на ручну мемоізацію** як на основний спосіб «оптимізації» — компілятор автоматично стабілізує значення й зменшує зайві ре-рендери там, де це безпечно.

Деталі реалізації залежать від офіційного релізу й конфігурації у проєкті; цей документ задає **стиль коду**, сумісний з таким напрямом.

---

## За замовчуванням

- **Не** додавати `React.memo`, `useMemo`, `useCallback` «на всяк випадок» або «щоб діти не ре-рендерились» без вимірювання або чіткої причини (див. нижче).
- Писати **прості компоненти**: передбачувані props, без зайвих проміжних стабільних посилань «для deps».
- Уникати патернів, які заважають аналізу: **мутація** об’єктів у state, «приховані» side effects у рендері, надмірна динамічність структури залежностей.

Чим менше штучної мемоізації — тим **простіше** читати код і тим **швидше** безболісно увімкнути компілятор.

---

## Коли ручна мемоізація все ще доречна

Окремі **виправдані** випадки (узгоджувати з командою, не розпорошувати по коду):

1. **Виміряна проблема продуктивності** (профіль, зависання списку з тисячами рядків тощо).
2. **Дорога чиста обчислення** (великі масиви, складний reduce) — інколи залишають `useMemo` після профілювання; компілятор з часом може закрити й це, політика залежить від версії tooling.
3. **Контракт сторонньої бібліотеки**: мемоізовані колбеки передаються в компонент, який порівнює за посиланням і без цього дає некоректну поведінку — краще ізолювати в обгортці, ніж розмазувати `useCallback` скрізь.
4. **Публічні стабільні ref-колбеки** для низькорівневих API (рідко в прикладному коді).

---

## Що допомагає і без компілятора

- Розбивати великі компоненти — менший blast radius ре-рендеру.
- Дані з сервера через TanStack Query — менше ручного sync і зайвого стану.
- Керовані форми та явні стани екрана — менше випадкових перерахувань.
