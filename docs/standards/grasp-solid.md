# GRASP, SOLID та чистий код

Принципи розподілу відповідальностей між модулями (**компонент, хук, feature, сторінка, `shared/api`**). Вони доповнюють [архітектуру Page-First](../architecture/overview.md) і [Coding Guidelines](./coding-guidelines.md): архітектура каже *де* лежить код, GRASP/SOLID — *хто за що відповідає*.

> **Пріоритет для команди:** спочатку орієнтуйся на **GRASP** при розкладанні логіки по файлах і шарах; **SOLID** — як стислий чеклист; **чистий код** — щоденні дрібниці іменування та розміру функцій.

---

## GRASP (General Responsibility Assignment Software Patterns)

### Information Expert

**Хто володіє даними — той виконує операції над ними.**

- Маппінг DTO → UI-модель поруч із місцем, де ці дані споживаються (`pages/.../mappers`, `features/...`), а не «де попало».
- Валідація поля форми — там, де є доступ до значень і правил цієї форми.

### Creator

**Хто створює об’єкт або збирає складну структуру — той модуль і відповідає за фабрику/збірку.**

- Складений об’єкт фільтрів для API збирає модуль, який знає контекст екрана (сторінка або feature), а не голий утиліт без контексту.

### Controller

**Один вхідний шар координації для сценарію** (не «бог-компонент»).

- Сторінка (`pages/...`) координує: які `features` підключити, які callbacks прокинути; складна оркестрація — у хуку `useXxxScreen`, а не в десятку `useEffect` у розмітці.

### Low Coupling / High Cohesion

**Мало залежностей назовні, багато зв’язаного всередині модуля.**

- Купа дрібних імпортів «хрест-накрест» між сусідніми фічами — сигнал підняти композицію в `pages/*` або спільне в `shared/*` ([детально](../architecture/overview.md#як-працювати-коли-одна-feature-сильно-залежить-від-іншої)).

### Polymorphism

**Різні реалізації одного контракту замість великих `switch` по типу.**

- Рендер рядка таблиці / картки через варіанти компонента або мапу `type → Component`, а не роздування одного компонента умовами.

### Pure Fabrication

**Штучний модуль, якого «нема в предметній області», але він знижує зв’язність.**

- `shared/api`, `shared/query-keys`, адаптер до API — допустима «вигадка», щоб не тягнути транспорт у UI.

### Indirection

**Посередник між залежностями** (провайдер, HTTP-шар), щоб міняти реалізацію без каскадних правок.

### Protected Variations

**Точка розширення там, де варіанти поведінки змінюються.**

- Ізоляція роботи з мережею в `shared/api` захищає компоненти від змін endpoint’ів або клієнта (`fetch` / `ofetch`).

## SOLID (коротко для React/TS)

| Принцип | На практиці |
|--------|-------------|
| **S**ingle Responsibility | Один модуль/компонент — одна причина зміни (наприклад: список vs модалка створення). |
| **O**pen/Closed | Розширення через композицію / нові варіанти, а не безкінечне редагування одного гіганта. |
| **L**iskov | Підтипи не ламають контракт (props, хуки поводяться передбачувано). |
| **I**nterface Segregation | Вузькі props / типи замість «прокидаючи весь `user` всюди». |
| **D**ependency Inversion | UI залежить від абстракцій (хуки, callbacks), а не від конкретного `fetch` у файлі компонента. |

## Чистий код — орієнтири

Це не дубль Clean Code книжки, а **мінімум домовленостей** разом із рештою репозиторію:

- **Імена** відображають намір (`handleSubmit`, не `click1`); уникати неочевидних скорочень.
- **Функції короткі** з одним ясним результатом; глибока вкладеність — виносити в іменовані кроки.
- **Без дублювання** копіпасти бізнес-правил у трьох місцях — DRY у межах розумного.
- **Коментарі** — «чому», а не «що робить кожен рядок».
- **Помилки та edge cases** обробляти явно, а не ковтати мовчки.
- Типова імплементація правил стилю, імпортів, React — у [Coding Guidelines](./coding-guidelines.md).
