## Feature

**Назва:** Translation Bulk Edit Panel  
**Скоуп:** Масове редагування значень перекладів для вибраних ключів у межах одного проєкту.

## Опис

Фіча додає side panel для bulk-редагування перекладів. Користувач вибирає кілька ключів у таблиці перекладів, відкриває панель і застосовує зміну значення для конкретної мови одразу до всіх вибраних ключів.

## Use Cases

1. Як перекладач, хочу змінити значення для групи ключів одразу, щоб не редагувати по одному.
2. Як редактор локалізації, хочу бачити кількість вибраних ключів перед застосуванням bulk-операції.
3. Як користувач, хочу скасувати bulk-редагування до підтвердження змін.

## UI стани

- `idle` — нічого не вибрано, bulk action недоступна.
- `selection` — вибрано 1+ ключів, кнопка bulk edit активна.
- `panel_loading` — панель відкрилась, підтягуються потрібні дані.
- `panel_ready` — форма доступна для введення значення.
- `submitting` — disable кнопок, показ loading-state.
- `success` — панель закривається, таблиця оновлена, success feedback.
- `error_validation` — показ field errors у формі.
- `error_server` — загальне повідомлення про помилку.

## API

- `GET /api/v1/projects/{slug}/keys/` — джерело списку ключів (вибір з таблиці).
- `PATCH /api/v1/projects/{slug}/keys/{key}/translations/` — застосування змін до конкретного ключа.
- За bulk-сценарію виконується послідовність/батч мутацій для вибраних ключів з подальшою інвалідацією `translationKeyKeys.lists(projectSlug)`.

## Data / State

- Server state: TanStack Query (`translationKeyKeys.*`).
- Form state: `@mantine/form` локально у панелі.
- URL state: поточні `search/lang/status/page/limit/view` зберігаються без змін.
- Client state: глобальний store не використовується для даних перекладів.

## Constraints

- Не змінювати архітектурні інваріанти зі `docs/architecture/*`.
- Не дублювати server data у Zustand.
- Інвалідація кешу — точкова, без широкого `['projects']`.
- Помилки полів мапити через `getFieldErrors`.

## Edge Cases

- Користувач зняв весь selection перед submit -> submit заборонений.
- Частина ключів недоступна через race condition (видалені іншим користувачем) -> частковий fail з повідомленням.
- 401 під час submit -> обробка через interceptor (без локальної retry-логіки в UI).
- Вибрана RTL мова -> input у панелі має `dir="rtl"`.

## Acceptance Criteria

- Можна застосувати bulk-редагування до N вибраних ключів.
- Після успіху список оновлюється і показує нові значення.
- Поля форми показують серверні validation errors.
- UI явно покриває `loading/empty/error/success` стани.
