# Architecture Overview

Цей документ — відправна точка для розуміння архітектури. Читати перед будь-якими змінами до структури проєкту.

> **Для агентів:** архітектурні рішення зафіксовані. Змінювати стек, шари Page-First або правила імпортів без явного дозволу людини — заборонено.

---

## Що це і як працює

TMS Frontend — SPA (Single Page Application) на React. Спілкується з REST API бекенду через HTTPS. Бекенд може знаходитись на окремому домені — фронтенд налаштований для роботи через CORS.

Додаток розгортається у Docker-контейнері з nginx, який роздає статичні файли після `vite build`. Це забезпечує незалежність від хостингової платформи.

```
┌──────────────────────────────────┐
│         Docker Container         │
│  nginx → /dist (vite build)      │
└──────────────────────────────────┘
               │ HTTPS
┌──────────────────────────────────┐
│         Backend API              │
│  /api/v1/  (окремий домен)       │
└──────────────────────────────────┘
```

## Чому Page-First підхід

Page-First обраний як основна архітектурна методологія з таких причин:

- **Фокус на екранах** — продуктова логіка формується від сторінок і user flow, а не від абстрактних доменів
- **Локальна ізоляція** — кожна сторінка має власний мікромодуль (`components`, `hooks`, `queries`, `mappers`, `types`, `ui`)
- **Кероване перевикористання** — якщо код потрібен між сторінками, він виноситься у `features/*`
- **Передбачувані залежності** — кожен шар має чітку відповідальність і не залежить від сусідніх модулів

## Шари Page-First

Шари розташовані від найвищого рівня абстракції до найнижчого. Верхній шар може імпортувати з нижнього — але не навпаки.

```
app/        Провайдери, bootstrap, глобальні стилі. Точка входу додатку.
            Також тут зазвичай живуть:
            app/routes або app/routing — декларація маршрутів, router config, route guards.
            app/layouts            — layout-компоненти верхнього рівня (public/private, shell, page wrappers).
pages/      Сторінки маршрутів. Кожна сторінка — окремий мікромодуль, зазвичай: components/, hooks/, queries/, mappers/, types/, ui/.
            Кореневий файл сторінки — точка входу, яка композиційно збирає поруч розташовані частини.
            Сторінки не імпортують код одна одної (це сусідні модулі).
features/   Перевикористовувані між сторінками модулі з прикладною логікою (напр. feature/catalog/product/components/banner).
            Feature не імпортує сусідні feature і не залежить від шарів вище (`pages`, `app`).
shared/     Найнижчий інфраструктурний шар без бізнес-логіки.
            shared/api    — обгортка над fetch / ofetch, базова логіка запитів (токен, retry після refresh), типи transport-рівня, доменні API-функції та їх типи.
                           Рекомендація: генерувати типізований API-клієнт через OpenAPI TypeScript.
            shared/ui     — глобальні UI-примітиви, обгортки над бібліотекою компонентів, або відсутні базові компоненти.
            shared/config — глобальні конфіги, константи, env-похідні налаштування, runtime/build-time параметри.
            shared/hooks  — загальні абстрактні хуки без прив'язки до конкретної бізнес-фічі.
            shared/lib    — дрібні утиліти, адаптери, pure helpers, форматери, guards, функції трансформацій без UI/HTTP-залежностей.
            shared/query-keys — фабрики ключів TanStack Query по доменах (єдине місце, без дублікатів і крос-імпортів між pages/features).
            shared/i18n   — ініціалізація i18n, словники, правила локалізації, fallback-стратегія.
```

**Правило імпортів:** `app` → `pages` → `features` → `shared`. Зворотні імпорти заборонені.

Додаткові обмеження:
- `pages/*` не імпортує інші `pages/*`
- `features/*` не імпортує інші `features/*`
- `shared/*` не залежить від `features/*`, `pages/*`, `app/*`
- все перевикористання між сусідніми модулями виноситься вниз по шарах

Enforcement: `eslint-plugin-boundaries`.

### Як працювати, коли одна feature сильно залежить від іншої

Якщо здається, що `feature/A` "потребує" `feature/B`, це сигнал підняти композицію вище:

- **Не робимо** прямий імпорт `feature -> feature`
- **Робимо** orchestration у батьківському модулі (`pages/*` або контейнер вище)
- **Прокидаємо залежності** через `props`, callbacks або render-props
- **Виносимо спільне** у нижчий шар (`shared/*` або окрема більш атомарна `features/*`, якщо є бізнес-сенс)

Приклад підходу:
- сторінка каталогу рендерить картку товару
- окремо підключає сценарій "додати в обране"
- у картку передається `onAddToFavorite(productId)` через props
- картка не знає про реалізацію feature "favorite", вона лише викликає контракт

Це зберігає ізоляцію модулів, зменшує зв'язність і спрощує тестування.

## Ключові архітектурні рішення

Нижче — узгоджений набір рішень для всього фронтенду. Деталі реалізації, приклади коду й edge cases — у профільних документах (посилання в кожному підрозділі та в [Frontend Architecture](./frontend-architecture.md)).

### Стан додатку

Розділення залишається таким: **server state**, **form state**, **URL state**, **client state** — це коректна модель; кожен тип має своє джерело правди.

- **Server state** — TanStack Query. Усі звернення до бекенду за даними та їх змінами — через **queries / mutations** (`useQuery`, `useMutation`), а не «напряму з компонента в HTTP-клієнт».
- **Form state** — локально (наприклад `@mantine/form`) у межах форми / екрану.
- **URL state** — TanStack Router (search params) для того, що має бути в посиланні (фільтри, сторінка, режим перегляду).
- **Client state** — локальний React-стан (`useState` / `useReducer`) для всього, що не сервер і не URL. Якщо той самий клієнтський стан потрібен **багатьом** споживачам — **React Context (провайдер)** з вузьким scope і мемоізованим `value`; **Zustand як типовий вибір не рекомендується** (лише після явного обґрунтування / окремого рішення).

Серверні дані **не дублюються** у глобальному клієнтському store: Query лишається джерелом правди для відповіді API.

У контексті Page-First:

- дані, специфічні для одного екрана, зазвичай оголошуються в `pages/<...>/queries` (або поруч зі сторінкою в її мікромодулі); **фабрики `queryKey`** — лише в `shared/query-keys/` (див. [State Management](./state-management.md#query-key-factory));
- коли той самий сценарій даних потрібен на кількох сторінках — query/hooks виносяться в відповідний `features/*`;
- глобальні речі без продуктової семантики (наприклад, лише технічні хелпери для кешу) — лише як виняток і вузько в `shared/*`.

Детальні політики, query configuration, auth flow — у [State Management](./state-management.md).

### Дані та HTTP

- Увесь транспорт до бекенду зосереджений у `shared/api`: одна тонка обгортка над **нативним `fetch`** або **`ofetch`** (рекомендовано замість `axios` для меншої поверхні та кращої узгодженості з екосистемою fetch). Інтерсептор-подібна логіка (токен, `401` / refresh, базовий URL) живе в цій обгортці, а не в компонентах.
- З компонентів викликаються не «сирий HTTP», а **функції, передані в `queryFn` / `mutationFn`** TanStack Query (інколи через невеликі обгортки-хуки в `pages/*/queries` або `features/*`).
- Маппінг DTO → модель для UI (`mappers`) тримається поруч із споживачем: у `pages/*/mappers` або в `features/*`, а не розмазується по випадкових компонентах.
- Компоненти не містять сирого виклику HTTP-клієнта і не дублюють транспортну логіку.

Контракти endpoint’ів і формати помилок — у [API Contracts](./api-contracts.md) та [API Reference](./api-reference.md).

### Аутентифікація

Access token зберігається **тільки in-memory** (наприклад React Context / провайдер + при потребі синхронний «міст» для HTTP-шару — див. [State Management](./state-management.md#auth-flow)), refresh token — у httpOnly cookie. При 401 — шар `shared/api` виконує refresh і повтор запиту. Cross-tab logout — через BroadcastChannel.

Повна схема auth flow, promise queue, silent refresh — у [State Management](./state-management.md#auth-flow).

### Routing

TanStack Router з type-safe маршрутами. Оголошення маршрутів і прив’язка `path` → сторінка зазвичай живуть у `app/routes` або `app/routing`, layout’и верхнього рівня — у `app/layouts`.

Auth guard реалізовано через `beforeLoad` на layout-маршруті (наприклад `_authenticated`) — не через розкидані перевірки всередині кожного екрану.

### Композиція екранів

Сторінка (`pages/*`) — місце збірки: вона підключає `features/*`, прокидає callbacks і дані через props, вирішує конфлікти між сценаріями. **Не** робимо прямих залежностей `feature → feature` (детальніше — вище в цьому документі).

Патерни UI-станів (`loading` / `empty` / `error`) — у [UI Patterns](../standards/ui-patterns.md).

### i18n

UI підтримує дві локалі: `en` та `uk`. Всі переклади зберігаються в одному файлі на локаль. RTL (`dir="rtl"`) встановлюється на input-ах для мов `ar`, `he`, `fa`, `ur` — визначається з API, не хардкодиться.

Ініціалізація та конфігурація — у `shared/i18n` (див. опис шару `shared` вище).

## Технологічний стек

| Категорія | Рішення | Чому |
|-----------|---------|------|
| Фреймворк | React 19 + Vite 6 + TypeScript 5 strict | Швидка збірка, типобезпека |
| UI | Mantine 8.x | Повне покриття UI-потреб, вбудована форма, модалки, нотифікації |
| Routing | TanStack Router v1 | Type-safe routes, вбудований auth guard через `beforeLoad` |
| Server state | TanStack Query v5 | Стандарт для REST: кеш, інвалідація, devtools; усі зміни з бекенду через queries/mutations |
| Client state | React Context (провайдери), локальний стан | Глобальний клієнтський стан — вузькі провайдери; Zustand не рекомендується як дефолт |
| HTTP | `fetch` + обгортка або `ofetch` | Менше залежностей, єдина модель з Web API; централізована логіка токенів / помилок у `shared/api` |
| Тести | Vitest + RTL + MSW / Playwright | Нативна інтеграція з Vite, реалістичний мокінг |
| Лінтинг | ESLint 9 + eslint-plugin-boundaries | Enforcement Page-First import rules |

## Розгортання

```dockerfile
# Збірка
FROM node:22-alpine AS builder
WORKDIR /app
COPY . .
RUN npm ci && npm run build

# Runtime
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
```

Конфігурація через змінні середовища у `.env`:

```
VITE_API_BASE_URL=https://api.yourdomain.com
```

> `VITE_*` змінні вбудовуються у білд на етапі збірки. Для зміни URL потрібен новий білд.
