## Що це за продукт

**TMS (Translation Management System)** — SPA web-додаток для централізованого керування перекладами в межах одного продукту або його середовища.

Система організовує переклади у вигляді ключів, де кожен ключ:
- має значення для різних мов
- може містити опис (контекст для перекладача)

Підтримується базова мова (source language), яка може використовуватися як fallback, якщо переклад відсутній.

### Основні можливості

- Організація перекладів по проєктах і мовах
- Редагування перекладів прямо у списку (inline editing)
- Розширене редагування через side panel
- Пошук і фільтрація: за мовою, за статусом (untranslated / translated), за ключем або значенням (повнотекстовий пошук)
- Масові операції (bulk видалення ключів)
- Експорт перекладів у JSON: flat та nested

### Режими редагування

**1. Inline редагування** — зміна значення перекладу безпосередньо у списку ключів, без переходу на інші екрани.

**2. Розширене редагування (side panel)** — бокова панель з доступом до всіх мовних варіантів, опису ключа та можливих додаткових функцій і налаштувань.

Зміни не зберігаються автоматично — використовується явне збереження (Save) з попередженням про незбережені зміни.

## Статуси перекладів

Статус перекладу не зберігається явно.

Він визначається на основі значення:
- відсутнє значення або порожній рядок "" → untranslated
- непорожнє значення → translated

Це derived state, який використовується для фільтрації та відображення прогресу локалізації.

### Обмеження поточної версії

- Відсутня система версій перекладів
- Немає ролей і workflow (review, approval)
- Відсутня історія змін і audit log
- Імпорт перекладів та інтеграції — в планах

### Аудиторія

- **Перекладачі** — для роботи з текстами
- **Розробники** — для керування ключами та експорту локалізацій

Система не є публічною і вимагає авторизації.

## Основні сутності

### Project
Контейнер верхнього рівня. Ідентифікується `slug`-ом (використовується в URL замість UUID).

### Language
Мова, яку підтримує проєкт. Одна мова — базова (еталон для перекладачів).

Бізнес-правила:
- Перша додана мова автоматично стає базовою
- Базову мову не можна видалити
- Не можна видалити останню мову проєкту
- Перед видаленням базової мови потрібно призначити іншу базовою

### TranslationKey
Ключ у dot-notation. Мінімум 2 сегменти: `auth.login.title`.

Бізнес-правила:
- Конфлікт вкладеності: якщо є ключ `menu`, не можна створити `menu.file` — і навпаки
Тобто ключі повинні забезпечувати можливість побудувати дерево, 
в якому кожен сегмент — це або ключ з перекладами, або проміжний сегмент.

### TranslationValue
Значення ключа для конкретної мови. Порожній рядок `""` або відсутнє значення = не перекладено.

### User
Користувач системи.

## Основні сценарії

### 1. Створення та налаштування проєкту
1. Створити проєкт (name + slug)
2. Додати мови (перша стає базовою автоматично)
3. Опційно — змінити базову мову

### 2. Робота з перекладами (основний workflow)
1. Перейти до проєкту → список ключів
2. Пошук/фільтрація ключів (за мовою, untranslated, за пошуковою фразою)
3. Inline або side-panel редагування значення перекладу
4. Зберегти зміни (Save) або скасувати (Cancel)

### 3. Створення ключів
1. Ввести ключ у dot-notation (`feature.component.label`)
2. Опційно — одразу заповнити переклади для кількох мов
3. Система перевіряє конфлікт вкладеності
4. Зберегти ключ і переклади (якщо вказані)

### 4. Видалення ключів
1. Вибрати один або кілька ключів
2. Підтвердити видалення

### 5. Експорт перекладів
Розробники завантажують переклади у JSON для підключення у свої додатки (flat або nested формат).

### 6. Управління користувачами
- Можна переглядати список користувачів
- Кожен користувач може редагувати свій профіль та змінити пароль
